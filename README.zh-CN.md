# 项目说明：德国 Freelancer.com 上的 Fair Pay 与语言要求

[English](README.md) | **中文**

MSK 6 Research Project（Fairwork Pilots）。本仓库记录：从公开平台抓取德国相关项目、按 Fairwork **Fair Pay** 做近似打分，再描述德语 / 英语岗位要求与得分、议价指标之间的关系。

---

## 1. 项目在做什么

在零工 / 平台经济里，Fairwork 用一套原则评价平台是否公平。本项目只聚焦其中一条：

**Fair Pay（公平薪酬）**

并结合德国 Freelancer.com 上常见的 **德语 / 英语岗位要求**，做三件事：

1. 用公开项目预算，近似判断岗位是否达到德国最低工资线（Fair Pay 打分）。
2. 从岗位文本编码德语、英语要求水平。
3. 用项目级投标数据构造议价代理指标，并按语言水平做分组描述。

**研究问题：**  
在德国 Freelancer.com 样本中，岗位要求的德语 / 英语水平，与 Fair Pay 得分、议价代理指标分别呈现怎样的描述性关系？德语与英语各自的相关强弱如何？

本项目**不做因果推断**，结果只作样本内描述。

---

## 2. Fair Pay 在本项目里怎么落地

Fairwork 原则 1 大致包含：

| 条款 | 含义 | 本项目处理 |
|------|------|------------|
| **1.1** | 扣除工作相关成本后，活跃工时至少达到当地最低工资（或更高的行业集体协议工资） | 用公开时薪预算与德国法定最低工资比较，通过记 **1 分**，否则 **0 分** |
| **1.2** | 达到当地生活工资 | 德国 Fairwork 报告所用生活工资参考值低于法定最低工资，故本项目**不再单独加分**；实际门槛统一为最低工资 |

**本项目采用的阈值：**

> 平均时薪（欧元）≥ **13.90 欧元/小时** → `fair_pay_score = 1`，否则 `= 0`

说明：公开帖看不到真实成本、等待时间、平台手续费，也看不到是否按时足额付款。因此这是「**预算是否够线**」的近似评分，**不是**完整 Fairwork 平台审计。

## 3. 数据从哪里来、长什么样

| 项 | 内容 |
|----|------|
| 平台 | [Freelancer.com](https://www.freelancer.com/) 公开项目页 |
| 抓取 | [Apify](https://apify.com/) `freelancer-scraper` |
| 清洗结果 | `dataset/dataset.csv`（由 `dataset_extact_script.py` 从 `rawdata/` 原始 JSON 生成） |
| 样本量 | 按项目 `id` 去重后 **162** 条（德国相关抓取） |
| 发帖语言 | 主要为英语约 120、德语约 42 |
| 计价方式 | **固定总价（fixed）130 条**；**按小时（hourly）32 条** |

**计价方式说明：**

- **固定总价（fixed）**：整份工作给一个总预算区间（例如 500–800 欧元做完）。
- **按小时（hourly）**：按小时预算区间（例如 15–25 欧元/小时）。

Fair Pay 打分主要使用 **hourly**：能直接和 13.90 欧元/时比较。fixed 缺少可靠工时，本项目未对其打 Fair Pay 分。

数据集字段、局限与再生方式见 `file/dataset说明.md`。原始 JSON 字段见 `file/rawdata说明.md`。

**数据里没有的内容（影响后面能做什么）：**

- 单个投标人明细、成交价、中标者
- 雇主信息、平台手续费、付款记录
- 项目完成后的评价
- 工人真实语言能力与实际到手收入

## 4. 项目流程（做了什么）

整体入口：`main.py`（依次跑语言编码 → Fair Pay → 语言×得分 → 议价）。

```
rawdata/*.json
    → dataset_extact_script.py → dataset/dataset.csv
    → main.py
         ├─ 货币统一为欧元（currency_utils）
         ├─ 语言水平编码（language_scoring）
         ├─ Fair Pay 打分与相关描述（fair_pay_analysis）
         ├─ 语言 × Fair Pay 交叉描述
         └─ 议价代理指标 × 语言（bargain_analysis）
```

### 4.1 货币处理

预算、报价等先按汇率换算到欧元列，便于与德国最低工资比较。脚本：`currency_utils.py`。

### 4.2 语言水平编码

脚本：`language_scoring.py`。依据标题、描述、技能标签、发帖语言等，为每条项目编码：

| 变量 | 取值 | 含义 |
|------|------|------|
| `de_level` | 0 / 1 / 2 / 3 | 不需要德语 → 基础 A1/A2 → 流畅 B1/B2 → 良好 C1/C2 |
| `en_level` | 1 / 2 / 3 | 入门 → 中等/流畅 → 精通 |

**编码步骤：**

1. **规则匹配**：关键词 / 短语表（如 `fluent german`、`kein deutsch`）与技能标签（如 `German`、`English`）自动打分；多档命中时取较低要求档。
2. **AI 辅助**：任一侧规则无法判定时，调用 **DeepSeek**（`deepseek-v4-pro`），根据岗位公开文本输出水平与理由；理由前缀为 `[AI辅助]`。需环境变量 `DEEPSEEK_API_KEY`（见 `.env`）。
3. **发帖语言兜底**：例如发帖语言为 `en` 时，英语至少记入门档等。
4. **最终 fallback**：AI 仍失败时，德语默认 0、英语默认 1，理由标为 `[fallback]`。

全样本语言结果见：`results/02_language_vs_score/language_all.csv`。

### 4.3 Fair Pay 打分

脚本：`fair_pay_analysis.py`。

- 仅保留 `type = hourly`，且欧元预算列齐全的项目。
- 用预算上下限计算平均时薪 `hourly_avg_eur`。
- 与阈值 13.90 比较得到 `fair_pay_score`（0/1）。
- 另输出若干因素与得分 / 时薪的 Spearman 相关系数表（描述关联强弱，不作因果解释）。

可打分样本：**n = 28**（hourly 共 32，部分因预算/汇率字段不全未入分）。

输出目录：`results/01_fair_pay/`。

### 4.4 语言 × Fair Pay

在已打分的 28 条上，按 `de_level` / `en_level` 汇总通过率，并计算语言水平与 `fair_pay_score` 的 Spearman ρ。

输出目录：`results/02_language_vs_score/`。

### 4.5 议价代理指标

脚本：`bargain_analysis.py`。定义说明见 `file/议价能力.md`。

因没有单个投标与成交价，只能在**项目级**构造代理分数 `bargain_score`：

1. **价格位置 \(P\)**：平均报价 `bid_avg` 在预算区间 `[budget_min, budget_max]` 中的相对位置。
2. **竞争强度 \(C\)**：`log(1 + bid_count)`。
3. 对 \(P\)、\(C\) 做 z-score 后合成：  
   `bargain_score = 0.5 * z(P) - 0.5 * z(C)`  
   （报价相对越高、投标人数相对越少 → 分数越高。）

可计算样本：**n = 151**（需投标人数、平均报价、预算上下限齐全，且上限 > 下限）。

输出目录：`results/03_language_vs_bargain/`。

## 6. 局限性（较完整）

### 6.1 数据与样本

- **样本量有限**：清洗后仅 162 条；Fair Pay 有效样本仅 **28** 条 hourly。高德语 / 高英语档位单元格往往只有个位数，通过率或均值波动大，不宜外推。
- **平台与国家范围**：只覆盖 Freelancer.com、德国相关抓取窗口；不能代表德国全部零工平台或其他国家。
- **抓取时点**：来自若干次 Apify 抓取的公开列表，不是全量历史库；热门/正在招标项目可能被过度代表。
- **公开字段不全**：`local`（是否要求现场）大量缺失；无精确城市、无雇主类型。

### 6.2 Fair Pay 打分本身

- **未扣除成本**：Fairwork 1.1 要求「扣成本后」达到最低工资。本项目只用毛预算时薪，未计入软件、设备、税费、平台费、未付费沟通时间等。
- **未验证付款**：看不到是否按时、足额支付，也看不到合同与拒付纠纷。
- **固定总价未评分**：130 条 fixed 缺少可靠工时，无法稳定换算时薪，故 Fair Pay 结论**不能**推广到固定总价项目。
- **生活工资条款简化**：因参考生活工资低于法定最低工资，本项目等价于只做「是否 ≥ 13.90」的 0/1 分，与完整 1.1+1.2 双分结构不完全相同。
- **预算 ≠ 成交价**：`budget_*` 是雇主标出的区间；实际成交可能更低或更高。

### 6.3 语言编码

- 编码的是**岗位要求**（文本中写了什么），不是工人真实德语/英语能力，也不是是否母语者。
- **规则 + AI**：关键词可能漏判或误判；DeepSeek 对模糊描述也可能不稳定。同一岗位换模型或换提示词，档位可能变动。
- **默认规则**：英语常偏向「至少入门」、规则失败时德语默认「不需要」，可能压低或抬高某些档的比例。
- 发帖语言（`language`）≠ 能力要求：英语发帖也可能要求德语，需靠描述判断。

### 6.4 议价指标

- 仅为**项目级代理**：用「平均报价相对预算位置」和「投标人数」合成，**不是**工人个体议价结果，也不是成交价。
- **fixed 与 hourly 合表标准化**：总价与时薪量纲不同，虽经 z-score，跨类型比较仍需谨慎。
- 无投标时间线、无中标价，无法区分「竞争激烈但最终高价成交」等情况。

### 6.5 统计与解释边界

- 报告的 Spearman ρ、分组均值/通过率均为**描述性关联**。
- **相关 ≠ 因果**：不能据此说「提高德语要求导致更高 Fair Pay」或「英语要求导致更低通过率」。
- 未做多元回归、混杂控制（如技能类别、项目难度、雇主所在地等可能同时影响语言与预算）。

### 6.6 伦理与方法边界

- 仅使用公开招聘信息，未访谈工人，也未收集可识别个人数据；适合一学期项目，但缺少工人视角的一手证据。
- AI 标注依赖外部 API 与密钥；结果可复现性受模型版本与提示词影响。

## 7. 如何运行

依赖见 `requirements.txt`。建议在项目根目录：

```bash
pip install -r requirements.txt
```

如需重新从原始 JSON 生成 CSV：

```bash
python dataset_extact_script.py
```

跑完全部分析（语言编码若缺 `DEEPSEEK_API_KEY`，规则判不全的条目会走 fallback）：

```bash
python main.py
```

结果写入：

| `results/01_fair_pay/` | Fair Pay 得分、相关表、图、结论 |

| `results/03_language_vs_bargain/` | 议价分数、与语言交叉 |

更细的图表说明见各目录下的 `说明.md`；文字摘要见 `结论.md`。
